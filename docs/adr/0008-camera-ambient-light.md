# ADR-0008 — Camera-as-ambient-light-sensor (D1001 SC2356 via esp_video)

**Status**: Proposed (experimental — spike in progress). Stage 1 (integration gate) **passed**:
`esp_video` + the SC2356 driver download, configure (sdkconfig), and link under ESPHome on the
ESP32-P4; `p4_camera_light` compiles clean (`tests/test_camera_light.yaml`). Not hardware-tested;
stages 2 (XL9535 power seq / shared I2C), 4 (auto-exposure) and 5 (drive `cfg_brightness`) remain.

## Context

We want **adaptive backlight**: dim `LCD_PWM` to ambient light. The D1001 has **no
dedicated ambient-light sensor** (only camera, dual mic, 6-axis IMU, battery). The only
light-sensing hardware is the **MIPI-CSI camera** (Seeed BSP / product page).

Two hard facts gate the naive "just read the camera" idea:

1. **ESPHome cannot read this camera.** `esp32_camera` is **DVP-only** (requires parallel
   `data_pins`/`vsync`/`href`/`pixel_clock`); the D1001 camera is **MIPI-CSI**. The
   `camera` component is only an abstract base — no MIPI-CSI platform, no SC2356 driver, in
   2026.6.x. A "Modular Camera Framework with MIPI-CSI support" is landing upstream but is
   unreleased/unconfirmed for this sensor. So a **custom external component over esp-idf
   `esp_video` is required**.
2. **Auto-exposure defeats a raw light meter.** The SC2356 + P4 ISP run AEC/AGC/AWB by
   default → average frame luma auto-normalises to ~mid-grey regardless of ambient light.
   To recover a light reading we must either **lock exposure+gain** (manual AEC/AGC) and
   read average luma (∝ light), or keep AEC and **read back the settled exposure+gain**
   (bright → short exposure/low gain; dark → the inverse) as an inverse proxy. Both need
   sensor register access via V4L2 controls.

The user green-lit an **experimental** custom driver, accepting the CPU/power cost, because
a real MIPI-CSI SC2356 driver for the D1001 is reusable well beyond a light meter.

## Reverse-engineered hardware map (Seeed `reTerminal-D1001` BSP)

- **Sensor**: SC2356 (RAW8, native mode `MIPI_RAW8_1280x720@30fps`). Driver exists in
  `esp_cam_sensor/sensors/sc2356/` (Espressif `esp-video-components`).
- **SCCB (sensor config I2C)**: **I2C_0 = SCL GPIO38 / SDA GPIO37** (shared bus:
  touch + camera + light). ESPHome already drives the GSL3670 touch on this bus.
- **Power/reset via the XL9535 I2C expander** (NOT direct GPIOs):
  `CAM_EN = EXP1`, `CAM_PWDN = EXP3`, `CAM_RST = EXP9`. Must be sequenced (enable → power
  up → release reset) **before** `esp_video_init`.
- **MCLK**: generated by the P4 CSI/ISP peripheral (no discrete MCLK pin to drive).
- **sdkconfig** (from the factory firmware): `CONFIG_CAMERA_SC2356=y`,
  `CONFIG_CAMERA_SC2356_MIPI_RAW8_1280X720_30FPS=y`,
  `CONFIG_ESP_VIDEO_ENABLE_ISP_PIPELINE_CONTROLLER=y`,
  `CONFIG_ESP_VIDEO_ENABLE_DVP_VIDEO_DEVICE=n`. Managed component
  `espressif/esp_video ~1.2`.

## esp_video / V4L2 flow (from BSP `app_video.c`)

```
esp_video_init(&{ .csi = { .sccb_config = { reuse i2c_master_bus_handle OR init_sccb },
                           .reset_pin = -1, .pwdn_pin = -1 } })   // power seq done via XL9535
fd = open(ESP_VIDEO_MIPI_CSI_DEVICE_NAME, O_RDONLY|O_NONBLOCK)
ioctl VIDIOC_QUERYCAP / VIDIOC_G_FMT / VIDIOC_S_FMT           // pick RAW8 (native, cheapest)
ioctl VIDIOC_S_EXT_CTRLS  -> lock AEC/AGC, set exposure+gain  // OR G_EXT_CTRLS to read them
ioctl VIDIOC_REQBUFS / VIDIOC_QUERYBUF / mmap / VIDIOC_QBUF
ioctl VIDIOC_STREAMON
ioctl VIDIOC_DQBUF -> average bytes (RAW8 ≈ luminance) -> VIDIOC_QBUF   // one frame per poll
ioctl VIDIOC_STREAMOFF   // between reads, to save power
```

For a light meter we do **not** need the RGB/ISP pipeline or continuous streaming: grab one
**RAW8** frame every N seconds, average (a subsample of) the bytes, map to a 0..1 lux proxy.

## Decision (plan, incremental)

New external component `p4_camera_light` (name TBD), built in stages, each a compile gate:

1. **Integration gate** — `__init__.py` pulls `espressif/esp_video` via
   `esp32.add_idf_component` + sets the SC2356/ISP sdkconfig via `add_idf_sdkconfig_option`;
   C++ does `esp_video_init` + open + `VIDIOC_QUERYCAP` and logs the driver banner. Proves
   the stack links/boots under ESPHome on P4. **Highest-risk unknown — do first.**
2. **Power sequencing** — bring CAM_EN/PWDN/RST up via the XL9535 before init. Risk: the
   expander + the GPIO37/38 I2C bus are **already owned by ESPHome** (touch); we must reuse
   ESPHome's `i2c_master_bus_handle` (pass it into `sccb_config`, `init_sccb=false`) rather
   than let esp_video re-init the bus, and drive the expander through ESPHome's `xl9535`.
3. **Capture + average** — one RAW8 frame → mean luma → `sensor::Sensor` (0..1 or lux-ish).
4. **Auto-exposure handling** — lock AEC/AGC + fixed exposure/gain (Option A), or read the
   settled exposure/gain (Option B). Validate on hardware which tracks ambient light better.
5. **Consume** — an HA automation (or on-device) maps the sensor to `cfg_brightness`.

### Risks / open questions
- **I2C bus & XL9535 co-ownership** with ESPHome (touch already uses both) — the main
  integration risk. Must share handles, not double-init.
- **CPU/RAM/power**: spinning the CSI+ISP+DMA even briefly is heavy on the loop; capture off
  the main task, low cadence (e.g. every 30–60 s), stream off between reads.
- **esp_video under ESPHome's IDF version**: managed-component/IDF-version compatibility is
  unproven — gate #1 exists precisely to find out early.
- **Privacy/UX**: a live camera in a child's bedroom purely to meter light — acceptable only
  because capture is momentary, RAW8, never stored/streamed.

### Alternatives (kept as fallback if the spike stalls)
- HA-driven `cfg_brightness` from `sun.sun` / a room sensor / a day-night schedule — already
  possible today (the number is an HA entity), zero firmware.
